The Four-Step Problem-Solving Loop

Quick reference · based on Pólya, How to Solve It · keep this next to you while solving

The one rule: Most of your time goes before you write code. If you're typing syntax and you don't yet know the answer in plain words, stop and go back to step 1 or 2.
STEP 1

Understand the problem

Goal: be able to explain it to a friend, with examples.

  • Restate the problem in your own words.
  • What exactly is the input? What exactly is the output?
  • Write 3+ concrete examples by hand: input → output.
  • List the edge cases (empty, zero, one, negative, very large).
  • Resolve ambiguities (e.g. is the count 0-indexed or 1-indexed?).

If you can't write an example by hand, you don't understand it yet — and no code will.

STEP 2

Make a plan (pseudocode)

Goal: a step-by-step recipe in plain language, no syntax.

  • Solve one small example by hand and watch what you do.
  • Write those steps as pseudocode — plain English, indented.
  • Ask: "have I solved something like this before?"
  • Break a big problem into smaller sub-problems.

Pseudocode keeps syntax out of your head so all your attention is on the logic.

STEP 3

Carry out the plan (code it)

Goal: translate pseudocode, one line at a time.

  • Turn each pseudocode line into one or two lines of code.
  • Keep it simple and mechanical — clever comes later.
  • If a step is too hard to translate, the plan was too vague → back to step 2.
STEP 4

Look back (test & reflect)

Goal: prove it works, then learn from it.

  • Run the code against every example from step 1.
  • Trace the failing case by hand to find where reality diverges from your plan.
  • Read the error message — it usually names the file, line, and cause.
  • Ask: can it be simpler or clearer? What did I learn for next time?

This step is where "my code seemed not functional" gets diagnosed instead of guessed at.

Which medium for which step?

StepBest mediumWhy
1 UnderstandPen & paper / plain notesFree-form; draw examples & arrows fast
2 PlanPen & paper or plain textNo compiler nagging about syntax
3 CodeEditorAutocomplete, run, real feedback
4 Look backEditor + paper for tracingRun in editor; trace bugs on paper
