The four-step loop: how to not freeze

~12 minutes  ·  The single most useful habit for solving any coding problem

You wrote a Fibonacci function and it "seemed not functional." Here's the thing: the bug wasn't really in your code. It was in the order you did things. Almost every beginner does this:

problem statement  ───►  start typing code  ───►  hope

That skips the two steps where the actual problem-solving happens. Experienced developers run a loop instead — one that's been taught since 1945, when mathematician George Pólya wrote it down in a little book called How to Solve It. It works for math, for cooking, and for code.

The loop:  1) Understand  →  2) Plan  →  3) Code  →  4) Look back.  Most of your effort belongs in steps 1 and 2 — before you type code.

This answers your real questions directly. Pen/paper vs. digital? Steps 1–2 are thinking, so do them wherever you think best (paper or plain text) — away from the compiler. Pseudocode? That is step 2. Solving faster? Counterintuitive but true: slowing down for steps 1–2 makes you faster overall, because you stop writing code that can't work.

The loop, applied to your Fibonacci problem

Goal: write fib(n) that returns the nth Fibonacci number. Watch how much happens before any code.

STEP 1 · UNDERSTAND

Restate it, then make examples by hand

In your own words: "each number is the sum of the previous two." But there's an ambiguity that probably broke your code — where does it start, and is n counted from 0 or 1? Pin it down by writing examples by hand:

fib(0) = 0
fib(1) = 1
fib(2) = 1   (0 + 1)
fib(3) = 2   (1 + 1)
fib(4) = 3   (1 + 2)
fib(5) = 5   (2 + 3)

Edge cases to note now: n = 0 and n = 1 don't have "two previous" numbers — they're special.

This is the step you skipped. If you can't write fib(5) = 5 by hand, no code will produce it. These six lines are also your test cases for step 4 — for free.

STEP 2 · PLAN (PSEUDOCODE)

Write the recipe in plain words — no syntax

Solve fib(5) in your head and notice what you actually do: you keep two numbers and keep adding them. Write that as pseudocode:

if n is 0  → answer is 0
if n is 1  → answer is 1
otherwise:
    prev = 0, curr = 1
    repeat (n - 1) times:
        next = prev + curr
        prev = curr
        curr = next
    answer is curr

Notice: no semicolons, no function keyword, no brackets to get wrong. All your attention is on the logic.

STEP 3 · CODE

Translate the pseudocode line by line

Now syntax is just transcription. Each pseudocode line becomes a line of JavaScript:

function fib(n) {
  if (n === 0) return 0;
  if (n === 1) return 1;

  let prev = 0;
  let curr = 1;
  for (let i = 2; i <= n; i++) {
    const next = prev + curr;
    prev = curr;
    curr = next;
  }
  return curr;
}

If a pseudocode line is hard to translate, your plan was too vague — go back to step 2, don't improvise in code.

STEP 4 · LOOK BACK

Test against your hand-made examples

Run it against every example from step 1:

console.log(fib(0)); // expect 0
console.log(fib(1)); // expect 1
console.log(fib(5)); // expect 5

If one is wrong, trace it by hand: walk the loop for that input and find the exact line where reality diverges from your plan. That's debugging — not guessing, not re-typing and hoping.

That's the whole skill. The Fibonacci answer is disposable; the loop is the thing you keep.

Your turn (do this before the quiz)

Pick a fresh, tiny problem — e.g. "return the largest number in an array" — and run all four steps:

  1. On paper or in a plain text file, write 3 examples by hand (include an empty array).
  2. Write the pseudocode. Then open your editor.
  3. Translate it to JS and test against your 3 examples.

Try paper for steps 1–2 at least once — notice whether it frees up your thinking. There's no wrong answer; the goal is to feel the difference between thinking and typing. Keep the one-page loop reference open while you do it.

Check your understanding

1. What is the correct order of the four-step loop?

2. Why write input→output examples by hand before coding?

3. What is pseudocode, and which step uses it?

4. Your code gives a wrong answer for one input. Best next move?

5. Why do steps 1 and 2 belong away from the editor (paper/text)?
