What are generics, and why do they exist?

~10 minutes  ·  Interview relevance: high

Generics are TypeScript's answer to a simple problem: how do you write a function that works with any type, but still lets TypeScript catch your mistakes?

You've already used generics without knowing it. Every time you write Array<string> or Promise<number>, you're using them. This lesson explains what's actually happening.

The problem generics solve

Suppose you want a function that takes a value and returns it unchanged — the simplest possible function:

// JavaScript — no types
function identity(arg) {
  return arg;
}

In TypeScript, you need to give arg a type. Your first instinct might be any:

function identity(arg: any): any {
  return arg;
}

This compiles. But you've thrown away all type safety:

const result = identity("hello");
// TypeScript thinks result is: any
// You've lost the information that it's a string

Why any is dangerous in interviews: If you reach for any in an interview, the interviewer will push back. It signals you're opting out of TypeScript, not using it.

Enter generics: a type parameter

Generics let you say: "I don't know what type this will be yet — but whatever it is, track it."

function identity<T>(arg: T): T {
  return arg;
}

T is called a type parameter. Think of it like a variable — but for types, not values. When you call the function, TypeScript fills in T automatically:

const a = identity("hello");  // T = string  → returns string
const b = identity(42);        // T = number  → returns number
const c = identity(true);      // T = boolean → returns boolean

Key insight: TypeScript infers T from the argument you pass. You rarely need to write it explicitly.

Why the name "T"? interview

T is just a convention — short for "Type." You can name it anything. But in interviews, using T is the expected convention. You'll also see:

// Multiple type parameters — also conventional names:
function pair<T, U>(first: T, second: U): [T, U] {
  return [first, second];
}

// For collections, K (key) and V (value) are common:
function getProperty<T, K extends keyof T>(obj: T, key: K) {
  return obj[key];
}

Don't worry about extends keyof yet — that's Lesson 3. Just notice the naming pattern.

The angle bracket syntax interview

The angle brackets <T> declare that this function has a type parameter. They appear:

// 1. After the function name (declaration)
function identity<T>(arg: T): T { ... }

// 2. On arrow functions — slightly different placement
const identity = <T>(arg: T): T => arg;

// 3. Explicitly passing the type (rarely needed, but you may see it)
identity<string>("hello");  // forces T = string

Arrow function gotcha: In .tsx files, <T> at the start of an arrow function looks like JSX. TypeScript sometimes gets confused. You'll often see <T,> (with a trailing comma) to disambiguate. You don't need to memorize this for most interviews — just know it exists.

What you can do with T inside the function

The constraint is important: inside the function, you can only use T in ways that work for any type. TypeScript will stop you if you try to use type-specific operations:

function identity<T>(arg: T): T {
  return arg.toUpperCase(); // ❌ Error! T might not be a string
}

function identity<T>(arg: T): T {
  return arg; // ✅ Works for any T
}

This is the trade-off: generics give you flexibility, but only within the constraints of what all types share. You'll learn to unlock more capabilities with extends in the next lesson.
