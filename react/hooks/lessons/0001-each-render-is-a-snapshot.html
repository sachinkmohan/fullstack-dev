Lesson 1  ·  React Hooks  ·  Mental models

Each render is a snapshot

~7 minutes  ·  One concept only: state values are frozen per render

You've used useState already. This lesson doesn't teach what it does — it teaches the one mental model that makes every other hook stop being weird.

Start with a puzzle

Interview classic

What does this button show after one click?

function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    setCount(count + 1);
    setCount(count + 1);
    setCount(count + 1);
  }

  return <button onClick={handleClick}>{count}</button>;
}

Most people guess 3. The answer is 1.

This trips up nearly every intermediate React engineer in an interview. Once you can explain why, you've unlocked the mental model that explains almost every hook gotcha you'll ever hit.

The one thing to remember

A component is just a function. It gets called over and over. Each call takes a frozen snapshot of state, and that snapshot never updates during the call — even if you call setState ten times.

That's it. The whole lesson.

What "frozen snapshot" really means

When React calls your component, it hands you count as a number — not a reference to a live value. Inside that call, count is a constant.

So when you wrote:

function handleClick() {
  setCount(count + 1);  // count is 0, so this is setCount(1)
  setCount(count + 1);  // count is still 0, so this is also setCount(1)
  setCount(count + 1);  // count is still 0, so this is also setCount(1)
}

You didn't add 1 three times. You scheduled "set to 1" three times.

React then re-runs your component. On the next call, count is a new constant — this time, it's 1.

render #1
count = 0
render #2
count = 1
render #3
…

Each render has its own count. Its own handleClick. Its own everything. They don't share variables — they're separate calls to a function, each with their own local scope.

The mental flip

If you've written class components, this is the unlearn: this.state.count was a live property on a long-lived object — it changed under your feet. useState's count is a value, captured in a closure that belongs to one specific render.

If you've never written class components: even better. Just remember — state isn't a variable you mutate. It's a value React hands you, frozen for the duration of this render.

Why interviewers love this

Because the snapshot model is the root cause of:

· Stale closures inside useEffect (the effect captured an old snapshot)
· Why useEffect has a dependency array (to detect when the snapshot changed)
· Why setCount(c => c + 1) works when setCount(count + 1) doesn't (the updater function reads the latest value, not your frozen one)
· Why useRef exists (the escape hatch when you need something not frozen)

Each of those is a lesson on its own. They all start here.
