The dependency array

~6 minutes  ·  One concept only: the dep array is a list of values React compares to decide whether to re-sync

Lesson 4 reframed an effect as synchronization with an external system. This lesson answers the natural next question: how does React decide when the external thing has drifted and needs re-syncing? The answer is the second argument — the dependency array.

What React actually does

After every render, before running an effect, React performs one tiny check: it compares this render's dependency array against the previous render's, value by value, using Object.is. If every value is the same, it skips the effect. If any value differs, it runs cleanup (if any) and then runs the effect again.

next 'general' 42

All equal → skip the effect

prev 'general' 42
next 'random' 42

One differs → cleanup, then re-run the effect

The one thing to remember

The dependency array is not a list of "things that trigger the effect." It's a list of every reactive value the effect reads — so React can tell whether those values changed since last time. You don't choose what goes in it; the effect's body does.

The three signatures

useEffect(fn);          // no array → runs after EVERY render
useEffect(fn, []);      // empty → runs once (nothing to drift)
useEffect(fn, [a, b]);  // runs when a or b changes since last render

Notice that all three are the same rule. With no array there's nothing to compare, so the check always "fails" and it runs every time. With [] there's an empty list that's always equal to itself, so it runs once and never re-syncs. The bracket list is just the general case.

Why the linter is strict

The react-hooks/exhaustive-deps rule insists that every reactive value the effect reads — props, state, or anything derived from them — appears in the array. People fight this rule by deleting deps to "make it run less." That's the bug, not the fix.

Remember Lesson 2: each render has its own frozen variables. If your effect reads roomId but you omit it from the deps, the effect keeps the roomId from the render where it last ran — a stale closure. The room changes, your connection doesn't. The linter is protecting you from exactly that.

Predict the bug

An effect subscribes using userId, but the array is []. The user logs out and a different user logs in. What does the subscription do?

It stays subscribed as the old user forever — [] told React "this never depends on anything," so it never re-syncs. The fix isn't to silence the linter; it's [userId].

The mental model

Think of the dep array as React asking: "Is anything this effect cares about different from last time?" Your only job is to answer honestly — list everything the effect reads. If re-running too often is the problem, the real fix is to change what the effect depends on (Lessons 7, 11–13), not to lie in the array.

Why interviewers love this

"Why does the linter want me to add this dependency?" is a daily-life question that doubles as a senior signal. The weak answer is "to avoid warnings." The strong answer connects it to the snapshot model: "each render closes over its own values; an omitted dependency means the effect runs with stale ones, so the dep array is how React knows the captured values are still current." The classic follow-up — "what if adding the dep makes it loop?" — sets up Lessons 11–13 on referential equality.
