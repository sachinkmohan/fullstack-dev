You might not need an Effect

~6 minutes  ·  One concept only: if no external system is involved, the effect is probably the wrong tool

Lessons 4–6 built up how effects work. This one is the payoff — and the single most valuable interview page React publishes. The skill: looking at code and noticing an effect that shouldn't exist.

The one test that decides it

Recall Lesson 4: an effect synchronizes with an external system — the network, the DOM, a subscription, a third-party widget. So before writing any effect, ask one question:

"Is there an external system involved?" If yes — keep the effect. If no — you're just reacting to a render, and there's a better tool: compute during render, or run code in the event handler. Effects are for escaping React, not for chaining state.

Two whole categories of effect are unnecessary, and both come straight from that test.

Category 1 — Transforming data for rendering

If a value can be computed from existing props and state, compute it during render. Don't mirror it into separate state and sync it with an effect.

// ❌ redundant state + effect
const [fullName, setFullName] = useState('');
useEffect(() => {
  setFullName(first + ' ' + last);
}, [first, last]);

// ✅ just derive it
const fullName = first + ' ' + last;

The effect version renders once with the stale value, then re-renders after the effect corrects it — twice the work for a value you already had. There's no external system here, so the test says: no effect.

Category 2 — Handling a user event

If code should run because the user did something — clicked Buy, submitted a form — it belongs in the event handler, not an effect that watches state change.

// ❌ effect reacting to a flag
useEffect(() => {
  if (submitted) showToast('Order placed!');
}, [submitted]);

// ✅ do it where the event happens
function handleSubmit() {
  setSubmitted(true);
  showToast('Order placed!');
}

The toast is caused by a specific interaction, not by synchronizing with the outside world. Event-specific logic goes in the event handler; only logic caused by the component being shown belongs in an effect.

The diagnostic question

For any effect, ask: "What caused this code to run?"

· Caused by the component being displayed (it needs to connect, subscribe, fetch)? → effect is right.
· Caused by a particular interaction (a click, a submit)? → event handler.
· It's just data derived from props/state? → compute during render.

Spot the accidental effect

A component keeps const [visibleItems, setVisibleItems] and runs useEffect(() => setVisibleItems(items.filter(i => i.active)), [items]). Does it need the effect?

No. visibleItems is derived from items — there's no external system. Delete the state and the effect; write const visibleItems = items.filter(i => i.active) during render. (If the filter were expensive, you'd reach for useMemo — Lesson 12 — still not an effect.)

Why this matters beyond interviews

Every unnecessary effect is a source of bugs: extra render passes, stale-closure traps, and synchronization cascades where one effect's setState triggers another. Removing them makes code both faster and more correct. The interview framing is just the visible tip — this is the habit that separates fragile React from solid React.

Why interviewers love this

Showing a candidate a component with a needless effect and asking "anything you'd change?" is a favorite senior-screen move. The strong answer names the test out loud: "There's no external system here — fullName is derived from props, so I'd compute it during render and delete the state and the effect. Effects are for synchronizing with things outside React, not for reacting to other state." That demonstrates the mental model from Lessons 4–6 in action, not just recited.
