Effects as synchronization

~5 minutes  ·  One concept only: an effect synchronizes with an external system — it is not a lifecycle event

You've shipped useEffect already. This lesson doesn't teach the syntax — it replaces the mental model most people carry into it. That model is the single biggest source of effect bugs, and the single most common interview tell.

Start with a thermostat

A thermostat has one job: keep the room matching the number on the dial. It doesn't care when you set the dial, how many times you've touched it, or what the last value was. It only cares about one thing — does the room currently match the target? If not, it acts until it does.

It doesn't run "on mount." It doesn't run "on update." It runs whenever reality has drifted from the target, and stops when they agree.

The old model, and why it misleads

Most people learn effects as a translation table from class components:

componentDidMount    →  useEffect(fn, [])
componentDidUpdate   →  useEffect(fn, [deps])
componentWillUnmount →  useEffect(() => cleanup, [])

This works often enough to feel right, which is exactly the trap. It frames an effect as "code that fires at a moment in the component's life." So you start asking the wrong question: "when does this run?" — and you reach for an effect any time you want something to happen "after render."

The one thing to remember

An effect doesn't run at a moment. It keeps something outside React — a subscription, a connection, the document title, a timer — in sync with the props and state inside React. Ask not "when does this fire?" but "what am I keeping in sync, and with what?"

What "external system" means

An external system is anything React doesn't control: the browser, the network, a third-party widget, a websocket. If you're touching one of those from React state, that's a synchronization job:

useEffect(() => {
  const conn = createConnection(roomId); // sync TO this external thing
  conn.connect();
  return () => conn.disconnect();
}, [roomId]);                            // ...FROM this React value

Read it as: "keep an open connection that matches roomId." When roomId changes, the old connection no longer matches the target, so React tears it down and opens a new one. Same thermostat, new dial setting.

The reframed question

The lifecycle model asks "do I want this to happen on mount, or on update, or both?" The synchronization model asks two better questions:

· What external thing am I keeping in sync? (the connection, the title, the subscription)
· Which React values does it depend on? (those become the dependency array — Lesson 5)

Answer those two, and the effect almost writes itself. If you can't name an external system, that's a strong hint you don't need an effect at all — which is Lesson 7.

Why interviewers love this

"Walk me through useEffect" is a near-guaranteed question, and most candidates recite the lifecycle table. If you instead say "an effect synchronizes a React-external system with my props and state; I think about what I'm syncing and what it depends on, not about mount vs update" — you've signalled senior-level understanding in one sentence. The classic follow-up is "so when should you not use an effect?" — and now you have a principled answer: when nothing external is being synchronized.
