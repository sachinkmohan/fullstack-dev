useRef, the escape hatch

~5 minutes  ·  One concept only: a ref is a mutable box that survives renders without triggering them

Lesson 1 established that state is a snapshot — frozen per render, and changing it causes a re-render. Sometimes you want the exact opposite: a value that persists across renders but that you can mutate freely without causing a render. That's useRef.

What it gives you

const ref = useRef(0);

ref.current;        // read the box
ref.current = 5;    // write the box — no re-render

It returns a plain object with a single current property. React hands you the same object on every render and otherwise stays out of the way. Reading or writing current is just touching a normal JavaScript property — React never notices.

Render 1
count = 0
Render 2
count = 1
Render 3
count = 2
ref.current — one box, shared across all renders

State is a fresh snapshot each render. The ref is a single box that outlives every render.

The one thing to remember

State vs ref: changing state triggers a re-render and gives you a new snapshot; changing a ref does neither. Use state for anything the UI displays. Use a ref for values you need to remember but that the render output doesn't read.

The two jobs of a ref

1. Holding a mutable value across renders. A timer ID, the previous value of a prop, a "has this already run?" flag — things you read in handlers or effects but never render:

const intervalId = useRef(null);
function start() { intervalId.current = setInterval(tick, 1000); }
function stop()  { clearInterval(intervalId.current); }

2. Referencing a DOM node. Pass the ref to a JSX element's ref attribute and React fills current with the actual DOM node after render — your hatch to focus, measure, or scroll it:

const inputRef = useRef(null);
<input ref={inputRef} />
// later: inputRef.current.focus();

The rule that keeps it safe

Because writing a ref doesn't re-render, the screen won't update to reflect it. So: don't read or write ref.current during rendering (except for lazy first-time init). Touch it in event handlers and effects, where the timing is well-defined. If you find yourself wanting the UI to react to a ref change, that's your sign it should have been state.

Predict the bug

A developer stores a counter in a ref and renders {counter.current}, incrementing it in a click handler. They click three times. What shows on screen?

0 — the ref updates to 3, but writing it never triggered a re-render, so the displayed text is stuck at the first render's value. A value the UI displays must be state, not a ref. This is the mirror image of Lesson 1's snapshot lesson.

Why interviewers love this

"What's the difference between useState and useRef?" is a staple. The weak answer is "refs are for DOM access." The strong answer covers both halves: "A ref is a mutable box that persists across renders but doesn't trigger one when it changes, so its value isn't a snapshot. I reach for it for mutable values the UI doesn't display — timer IDs, previous values — and for DOM nodes. The moment the UI needs to reflect the value, it has to be state instead." That shows you understand refs as the deliberate opposite of the snapshot model.
