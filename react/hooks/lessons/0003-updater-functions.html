Updater functions

~5 minutes  ·  One concept only: how to read the latest state without the snapshot

You've seen the snapshot model freeze state per render, and you've seen what stale closures look like. This lesson covers the official escape: passing a function to setState instead of a value.

Two ways to call setState

setCount(count + 1);          // value form — uses snapshot
setCount(prev => prev + 1);   // updater form — uses latest

They look almost identical. The difference is which count they read.

The one thing to remember

The updater function receives the most recent state as its argument — whatever React has stored at the moment the update runs. Your closure's frozen count is never consulted.

The Counter, fixed

Recall the puzzle from Lesson 1:

function handleClick() {
  setCount(count + 1);
  setCount(count + 1);
  setCount(count + 1);
} // adds 1, because count is frozen

Switch to the updater form:

function handleClick() {
  setCount(prev => prev + 1);
  setCount(prev => prev + 1);
  setCount(prev => prev + 1);
} // adds 3

Now React runs each updater in order, passing in the result of the previous one: 0 → 1 → 2 → 3.

When to reach for it

Use the updater form whenever the new state depends on the previous state — and your handler or effect might not have the latest value. Three common situations:

· You're calling setState multiple times in a row (the Counter case).
· You're inside an effect or timer with a stale closure (Lesson 2).
· You're inside an async callback (a fetch then-handler, an event listener).

If the new state doesn't depend on the old, there's no benefit — setCount(0) and setCount(() => 0) are equivalent.

The mental model

The value form is a statement: "set count to this exact value."
The updater form is a recipe: "however many things are queued before me, take that result and add one."

React keeps a queue of pending updates. The updater form lets each update see the queue's current head — not the frozen value from when the handler was called.

Why interviewers love this

If you can articulate "updaters escape the snapshot by reading from React's queue, not from the closure", you're signalling a real mental model. The follow-up is usually: "When would the value form be safer than the updater form?" (Answer: when you want the value frozen on purpose — e.g., grabbing the count at click time even if later updates are pending. Rare, but it happens.)
