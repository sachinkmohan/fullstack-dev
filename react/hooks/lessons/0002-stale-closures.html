Stale closures

~7 minutes  ·  One concept only: functions remember the state from when they were created

You learned in Lesson 1 that each render is a snapshot. This lesson takes one consequence of that and makes it visible: any function defined during a render closes over that render's snapshot — and keeps it forever.

The classic interview bug

Interview classic

This counter is supposed to tick up once per second. What actually happens?

function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      setCount(count + 1);
    }, 1000);
    return () => clearInterval(id);
  }, []); // empty deps on purpose

  return <p>{count}</p>;
}

It shows 0, then 1, then 1, then 1 forever.

The interval keeps firing. But the count is stuck at 1.

The one thing to remember

The arrow function inside setInterval was created on the first render. It captured count = 0. On every tick, it still sees count = 0 — even though the real state has changed.

That's a stale closure.

Walking through it

On render #1, count is 0. The effect runs once (because deps are []) and creates an interval. The callback inside that interval is a closure — it has count = 0 baked into it.

One second later, the interval fires. It calls setCount(0 + 1). State becomes 1. React re-renders.

On render #2, there's a new count = 1, a new setCount, a new component scope entirely. But the effect doesn't re-run (deps are []). The old interval is still running. The old callback is still alive. It still sees count = 0.

So one second later it calls setCount(0 + 1) again. State is already 1. React notices no change. The screen never updates again.

Three ways to fix it

Fix 1 — Updater function. Don't read count from the closure. Read it fresh from React:

setCount(prev => prev + 1);

Fix 2 — Honest dependencies. Tell the effect it depends on count. The interval gets torn down and re-created every render — which works, but is wasteful:

useEffect(() => {
  const id = setInterval(() => setCount(count + 1), 1000);
  return () => clearInterval(id);
}, [count]);

Fix 3 — A ref. If you need a value that isn't snapshotted, useRef gives you a mutable box that always holds the latest. (We'll cover this fully in a later lesson.)

Why interviewers love this

Stale closures are the #1 source of "my React code is doing the wrong thing" bugs in real codebases. If you can spot one on a whiteboard and name all three fixes, you're showing senior-level intuition. The bug is also a back-door way to test if you really understand the snapshot model from Lesson 1.
