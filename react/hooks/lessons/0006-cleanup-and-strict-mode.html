Cleanup & Strict Mode

~6 minutes  ·  One concept only: cleanup undoes an effect, and Strict Mode runs setup→cleanup→setup to prove it

Lessons 4 and 5 covered what an effect syncs and when it re-syncs. This lesson is the other half of synchronization: the cleanup function — how an effect stops syncing. And it answers the question every React beginner eventually Googles: "why does my fetch run twice?"

The setup needs an undo

If an effect opens a connection, something must close it. If it adds an event listener, something must remove it. If it starts a timer, something must clear it. That "undo" is the function you return from the effect:

useEffect(() => {
  const conn = createConnection(roomId);
  conn.connect();                      // setup
  return () => conn.disconnect();      // cleanup — the undo
}, [roomId]);

React calls the cleanup in two situations: before re-running the effect (because a dependency changed), and when the component unmounts. Every setup is eventually paired with exactly one cleanup.

The one thing to remember

Cleanup isn't "code that runs on unmount." It's the inverse of your setup — it must fully undo whatever the setup did, so that running setup then cleanup leaves the world exactly as it started. Write the cleanup as the mirror image of the setup.

Why your fetch runs twice

In development, React's Strict Mode deliberately mounts each component, immediately unmounts it, then mounts it again. For your effect that means: setup → cleanup → setup. You see the effect run twice and panic.

What Strict Mode does to one effect in development:

setupeffect runs — connection opens
cleanupReact immediately tears it down
setupeffect runs again — the real one

This is not a bug and it does not happen in production. It's a test: if your effect survives an immediate setup→cleanup→setup with no leftover damage — no duplicate connection, no doubled-up listener — then your cleanup is correct. If running it twice breaks something, your cleanup is missing or wrong. Strict Mode is surfacing a real bug early, not creating one.

The fetch case, specifically

A fetch with no cleanup fires twice and you might get two responses landing out of order, the later one stomped by the earlier. The cleanup is an ignore flag (or an AbortController):

useEffect(() => {
  let ignore = false;
  fetchUser(userId).then(u => { if (!ignore) setUser(u); });
  return () => { ignore = true; };   // discard the stale response
}, [userId]);

Now even if two requests fly, the first one's result is ignored once its cleanup runs. The double-fire stops being scary the moment the cleanup makes it harmless.

Predict the behavior

An effect calls window.addEventListener('resize', onResize) and returns no cleanup. In Strict Mode, how many listeners are attached after the initial mount settles?

Two — setup ran twice with nothing removing the first. Every resize now fires onResize twice. The missing cleanup, removeEventListener, is the whole bug; Strict Mode just made it visible.

Why interviewers love this

"Why does an effect run twice in development?" is an extremely common screening question because the weak answer ("Strict Mode is buggy / I disable it") and the strong answer are miles apart. The strong answer: "Strict Mode double-invokes effects to verify my cleanup fully reverses my setup; if running setup→cleanup→setup leaves damage, my cleanup is wrong. It's a correctness check, and it doesn't run in production." That single sentence shows you understand cleanup as the inverse of setup, not as an unmount hook.
